| Strategy | What It Is | When & Why to Use It | Why You Should Avoid It |
| :--- | :--- | :--- | :--- |
| **1. Sentinel Errors** | Using a **specific global value** to represent an error (e.g., `if err == io.EOF`). | **When:** Rarely. Only for very well-known, stable, and low-level cases in the standard library. <br> **Why:** It provides a direct, simple check for a specific condition. | **Why Not:** Creates **tight coupling** between packages, making APIs brittle. It becomes part of your public API and prevents you from adding context to the error without breaking the check. |
| **2. Error Types** | Creating a custom **struct type** that implements the `error` interface (e.g., `if err, ok := err.(*MyError); ok { ... }`). | **When:** Sometimes, if the caller needs to access **structured data** from the error (like a line number or filename). <br> **Why:** Can carry more context than a simple value. | **Why Not:** Still creates **strong coupling** because the caller must import your package's custom type. This also makes APIs brittle. |
| **3. Opaque Errors** | Simply returning the error without inspecting its content. The caller only knows an operation succeeded or failed. | **When:** **This should be your default strategy** for most functions. <br> **Why:** Provides maximum flexibility for the function author to add context later without breaking the caller. Minimizes coupling between packages. | **Why Not:** In rare cases, the caller needs to know the **behavior** of the error (e.g., "Is this error temporary so I can retry?"). The opaque style alone doesn't allow this. |
| **4. Handle Errors Gracefully** (The `errors` package approach) | A best-practice approach using functions like `errors.Wrap` and `errors.Cause` from `github.com/pkg/errors`. | **When:** When you need to **add context** to an error as it bubbles up (e.g., annotating "open failed" to a low-level "file not found" error). Also, when you need to check the **behavior** of the original error (e.g., "Is it temporary?"). <br> **Why:** It combines the best of all worlds: you can add context without breaking checks, and you can unwrap the error to inspect its underlying cause or behavior. | **Why Not:** If you don't need to preserve the original error's identity or add stack trace information, a simple `return err` (opaque handling) might suffice. However, this approach is generally superior for most non-trivial applications. |

### Points to remember:
*   **Default to Opaque Errors**: Treat errors as a black box. Just return them.
*   **If You Must Inspect, Check Behavior, Not Type**: If you need to decide whether to retry an operation, check if the error **behaves** as temporary (e.g., has a `Temporary() bool` method), rather than checking what **type** it is.
*   **Add Context by Wrapping**: Use `errors.Wrap` to add context (like "read failed") to an error as it goes up the call stack. This provides a valuable trace for debugging.
*   **Handle an Error Once**: Log an error or return it, but **never do both**. Doing so creates duplicate log entries and loses context. Use wrapping to pass it up.
*   **Never Inspect `error.Error()` String**: The string output of an error is for humans (logs, screens), not for code logic. Relying on it makes your code fragile.
